PART A — REAL SPORTSBET ODDS WITHOUT A PAID API  (scrape the site's own JSON API)
================================================================================
Sportsbet's website is driven by an undocumented public JSON API (no key, no
login). Confirmed working endpoints (decimal odds), reference implementation:
github.com/declanwalpole/sportsbook-odds-scraper (sportsbook_implementations/
sportsbet.py). Reproduce this logic in-repo; don't add the package as a dep.

A1. New module afl_bot/data/sportsbet_odds.py exposing:
        def fetch_sportsbet_odds(event_urls_or_ids: list[str]) -> dict[str, float]:
            """Return {leg_name: decimal_odds} in the SAME key format the report
            already uses, so it drops straight into odds_book (same contract as
            data/live_odds.py fetch_live_odds)."""

A2. Endpoints (GET, headers: Accept: application/json + a desktop User-Agent):
    event_id = the 7-digit number at the end of a Sportsbet match URL, e.g.
      https://www.sportsbet.com.au/betting/australian-rules/afl/<match-slug>-NNNNNNN
    (a) Market groupings for an event (the "card"):
        https://www.sportsbet.com.au/apigw/sportsbook-sports/Sportsbook/Sports/Events/{event_id}/SportCard?displayWinnersPriceMkt=true&includeLiveMarketGroupings=true&includeCollection=true
        -> json["marketGrouping"] = [{ "id":..., "name":... }, ...]
    (b) Markets within a grouping (the actual prices):
        https://www.sportsbet.com.au/apigw/sportsbook-sports/Sportsbook/Sports/Events/{event_id}/MarketGroupings/{market_group_id}/Markets
        -> [ { "name": market_name, "id":..., "selections": [
                 { "id":..., "name": selection_name,
                   "price": {"winPrice": <decimal_odds>},
                   "displayHandicap": <line or null> }, ... ] }, ... ]
    If a response's Content-Type isn't application/json, you're being geo-blocked
    (not in AU) or rate-limited — log it and fall back (do NOT retry-storm).

A3. Getting the event_ids for the round — AUTO-DISCOVER, no pasting (DEFAULT).
    Ben should NOT have to paste URLs. The bot already knows the round's fixtures
    (home/away teams from the Squiggle/fixture data it builds the report from);
    use that to find the Sportsbet events automatically:
      1. Fetch Sportsbet's Australian-Rules / AFL competition listing from the
         same apigw the per-event endpoints live on. Determine the exact route
         LOCALLY (you're on Ben's AU machine): open
         sportsbet.com.au/betting/australian-rules/afl, watch the Network tab for
         the apigw XHR it fires, and use that "competition events" / SportCard
         endpoint. It returns the round's events, each with an event id + the two
         team names. (Likely shape:
         .../apigw/sportsbook-sports/Sportsbook/Sports/Competitions/{afl_comp_id}/Events
         — confirm the live path rather than trusting this string.)
      2. Match each Sportsbet event to a round fixture by NORMALISED team name
         (afl_bot.data.teams.normalize_team_name on both sides). Keep the event
         id per matched fixture; log any fixture you couldn't match.
      3. Feed those event ids straight into the SportCard/Markets calls (A2).
    So a normal run is just:  round-report --year 2026 --round 16 --sportsbet
    with NOTHING to paste.
    - FALLBACK ONLY (optional, absent is fine): if auto-discovery can't match a
      fixture (endpoint changed, or match not posted yet), read override URLs
      from reports/{year}_r{round}_sportsbet_urls.json (["<url>", ...]) and take
      the trailing 7-digit id. Used only to patch a gap, not the normal path.

A4. Map Sportsbet markets/selections -> the bot's leg keys (cli.py builds them as
    f"{player_name} {line}+ {stat}", e.g. "Will Ashcroft 20+ disposals";
    "<Team> to win"; "Total points 170.5+"). Handle these AFL markets:
      * Head-to-head / Match Result      -> "<normalised team> to win"
      * Total Points (Over/Under)        -> "Total points <line>+"  (Over side)
      * Player Disposals (alt lines / "N+ Disposals" / Over N.5)
                                         -> "<player> <int>+ disposals"
      * Anytime Goalscorer / To Kick N+ Goals
                                         -> "<player> <N>+ goals"
      * Player Marks                     -> "<player> <N>+ marks"
      * Player Tackles                   -> "<player> <N>+ tackles"
    Line handling: a milestone market ("20+ Disposals") -> N=20. An Over/Under
    market with displayHandicap 19.5 -> the OVER selection = "20+" (ceil to the
    next integer, matching how the bot names lines). Take the posted winPrice.
    Normalise team names via afl_bot.data.teams.normalize_team_name and player
    names through the SAME identity/normalisation path cli.py uses, BEFORE
    returning, or the keys won't join and edges won't compute. Log how many
    selections matched a bot leg vs were dropped (low hit-rate must be visible).

A5. Caching + safety: cache each event's raw JSON ~2 min (reuse the data_cache
    pattern). One request per market grouping is fine; don't fan out further.
    Any network/parse/geo-block error -> return {} for that event and continue.
