BUILD: MULTIS DASHBOARD + BET TRACKER / P&L  (local "command center", like the
trade bot)

Date: 2026-06-26
From: Ben

Goal: a local browser dashboard for the same-game-multi results where Ben can
SEE each round's multis, TICK the ones he actually placed (with stake + the odds
he got), and the bot AUTO-SETTLES them after the round and tracks running P&L /
ROI / strike rate over the season. Same spirit as the trade-bot command center:
a local app launched from the CLI, data persisted to a file, nothing in the
cloud. Do NOT touch the sim/modelling, the odds scrape, or the ladder logic —
this is a presentation + tracking layer ON TOP of the existing report.

Build it in stages; keep pytest green; commit at the end. Runs on Ben's machine.

--------------------------------------------------------------------------------
PART A — Emit the multis as machine-readable JSON (so the UI can render them)
--------------------------------------------------------------------------------
When round-report runs, ALSO write reports/{year}_r{round}_multis.json next to
the .md. One record per rung, BOTH ladders, with a STABLE id so a placed bet can
be linked back. Suggested schema per rung:
    {
      "id": "2026-r16-HAW-GWS-model-2.10",   # year-round-game-ladder-band, stable
      "year": 2026, "round": 16,
      "game": "Hawthorn vs Greater Western Sydney",
      "ladder": "model" | "sportsbet",
      "band": 2.10,
      "legs": [ {"player": "...", "market": "disposals", "line": 20,
                 "name": "Will Day 20+ disposals", "book_odds": 1.42} , ... ],
      "model_joint": 0.48,
      "model_fair": 2.10,
      "book_combo": 1.41,          # product of legs' real Sportsbet prices (null if unpriced)
      "edge": -0.245,              # model vs book (null if unpriced)
      "value_pick": false
    }
Reuse the exact numbers already computed for the .md so the JSON and the report
never disagree. (Refactor render so both consume one in-memory structure.)

--------------------------------------------------------------------------------
PART B — Bets ledger (persisted to a file so it survives restarts)
--------------------------------------------------------------------------------
reports/bets_ledger.json — a list of placed bets Ben recorded:
    {
      "bet_id": "<uuid>",
      "multi_id": "2026-r16-HAW-GWS-sportsbet-3.00",   # links to PART A record
      "year": 2026, "round": 16, "game": "...", "ladder": "sportsbet",
      "legs": [ ... copied from the multi at placement time ... ],
      "stake": 25.0,
      "taken_odds": 3.10,        # the odds Ben ACTUALLY got (he types it; may differ from book_combo)
      "placed_at": "2026-06-26T13:00:00+10:00",   # AEST/AEDT (Australia/Melbourne)
      "status": "pending" | "won" | "lost" | "void",
      "settled_at": null,
      "payout": null,            # stake*taken_odds if won, 0 if lost, stake if void
      "leg_results": null        # per-leg hit/miss once settled
    }
Snapshot the legs at placement so later report regenerations can't mutate a
recorded bet. All times in Australia/Melbourne (matches the trade-bot dashboard).

--------------------------------------------------------------------------------
PART C — Auto-settlement (use the grading the bot ALREADY has)
--------------------------------------------------------------------------------
The bot already grades a completed round against actuals (cli `grade-round`,
player-stat actuals, reports/{year}_r{round}_predictions.csv). Add a settle step
that, for each PENDING bet whose round is complete:
  - For each leg, look up the player's ACTUAL stat for that game and test the
    line (e.g. disposals >= 20). Team/H2H and totals legs settle off the result.
  - Won only if EVERY leg hit; else lost. (No partial.) Void a leg only if the
    player didn't play / market voids — then treat the multi per standard book
    rules: drop the void leg and re-settle on the rest (document this choice).
  - Set status, settled_at, payout (stake*taken_odds on win, 0 on loss), and
    leg_results.
Expose as `python -m afl_bot.cli settle-bets [--year --round]` AND call it
automatically when the dashboard loads / on a "Settle now" action. Reuse the
existing actuals path — do NOT build a second stats source.

--------------------------------------------------------------------------------
PART D — The dashboard (local web app, launched from the CLI)
--------------------------------------------------------------------------------
A small local Flask (or stdlib http.server) app in afl_bot/dashboard/, launched
by `python -m afl_bot.cli dashboard` -> opens http://127.0.0.1:8765 . Single-page
dark "command center" styling to match the trade bot. Endpoints read/write the
JSON files from PART A/B (no external DB). Pages/panels:

  1) ROUND VIEW — pick a round (dropdown, defaults to latest with a multis.json).
     Each GAME is a card; inside, the two ladders (Model / Sportsbet) as tables:
     legs, band, model %, model fair, book combo, edge, VALUE PICK highlighted.
     Sort/filter controls: by edge, by price (band), value-picks-only toggle.
     (This is the surface Ben places from — needed even though the headline ask
     is the tracker.)
  2) PLACE A BET — on any rung, a "I placed this" action opens a small form:
     stake + taken_odds (prefilled with book_combo, editable), -> POST appends to
     the ledger as pending. Show a confirmation.
  3) TRACKER / P&L PANEL — the headline. Show:
       - Open (pending) bets, and settled bets grouped by round.
       - Per-bet: legs, stake, taken odds, status, payout, profit (payout-stake).
       - Season summary: total staked, total returned, NET PROFIT, ROI %,
         strike rate (won/total settled), and a simple cumulative-profit line
         (Chart.js from CDN is fine).
     A "Settle now" button runs PART C for completed rounds and refreshes.
  4) Optional REFRESH: a button that shells `round-report --sportsbet` for the
     current round and reloads (nice-to-have; fine to skip if fiddly).

Persistence: write straight to reports/bets_ledger.json on every change (server
side) so it survives restarts and is git-trackable. All datetimes Australia/
Melbourne. Keep the whole UI in ONE html/template + the small server; no build
step.

--------------------------------------------------------------------------------

- ledger: placing a bet appends a pending record with a snapshot of the legs.

- P&L summary: net profit / ROI / strike rate computed correctly over a small

- Don't require a live server in the test suite — test the data/settlement layer
  directly and (optionally) the Flask routes with the test client.
- pytest -q green. Commit. README: how to launch (`... dashboard`), how settle
  works, where the ledger lives. Update memory.

cards, ticks the ones he placed with his stake + odds, and after the round the
